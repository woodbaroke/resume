
* 数据链路层
	* 隐藏物理层细节——实现的 IP地址与机器物理MAC地址的转化 ，
	* ARP（地址解析协议）    IP->MAC 
		* 主机向所在网络广播一个ARP请求，请求包含目标机器的网络地址。网络所有机器都收到请求，但只有目标机器回应一个ARP应答，包含自己的物理地址。
		* ARP维护一个高速缓存。linux系统命令可以查看
			* arp-a 查看arp缓存
			* arp-d 删除某个缓存
			* arp-s 增加某个缓存
		
	* RARP (逆地址解析协议）MAC->IP
* 网络层
	* 隐藏链路拓扑结构——通过路由器节点选择实现两IP通信，对上层来说两个机器是直接相连的
	* 采用IP协议进行路由选择，ICMP协议（因特网控制报文协议）来检测网络连接 ping程序就是用此
		* IP分片——当IP数据报长度超过帧的MTU，将被分片传输，到最终目标机器才重组。
			* 以太网帧的MTU是1500，IP头部20字节，因此IP数据报只能1480字节
			* IP头部有三大信息
				* 数据报标识，相同ip数据报标识不变
				* 有不同的位偏移
				* 除最后一个分片外，其他分片设置MF标志
		* IP路由机制
			* route 、 netstat 命令可以查看路由表
			* IP转发机制
				* 检测TTL如果为0丢弃
				* 检测数据报IP地址是否为本机，如果不是发送一个ICMP选路失败报文发送端。
				* 给源端发送一个ICMP重定向报文
				* TTL-1
				* 处理IP头部选项
				* 如有必要，执行IP分片
			* ICMP  重定向报文 发送两个信息
				* 引起重定向的IP数据报
				* 应该使用的路由IP地址
* 传输层
	* TCP、UDP特点及区别
		* TCP协议 提供可靠 面向连接 基于流的服务。 
			* 可靠——超时重传、数据确认。
			* 面向连接——建立TCP连接。
			* 基于流——流的数据没有长度限制，源源不断的写 另一边不停读。
		* UDP协议 不可靠、无连接、面向数据报的服务
			* 不可靠——中途丢失或校验数据错误直接丢弃
			* 无连接——每次发送消息都得指明地址
			* 基于数据报——每个UDP数据报有长度，接收端必须以该长度为最小单位将消息一次性读出
	* TCP细节 ——tcpdump可以抓取报文段
		* TCP头部 最大60字节
			* 固定头部 20字节
				* 16位端口号——源端口和目的端口
				* 32位序号——一次TCP传输中某一方向的字节流编号
				* 32位确认号——对对方的TCP报文段的相应，为序号值+1
				* 4位头部长度——标识该TCP头部有多少个4字节，4位最大15，TCP头部最长60字节
				* 6位标志位
					* URG——紧急指针是否有效
					* ACK——确认号是否有效，带ACK标志的TCP报文段被称为确认报文段
					* PSH——提示接收方立即从TCP接收缓冲区读取
					* RST——要求重新建立连接，携带者为复位报文段
					* SYN——请求建立连接，携带者为同步报文段
					* FIN——关闭连接，携带者为结束报文段
				* 16位窗口大小——接收通告窗口，告知缓冲区还能容纳多少字节，来TCP流量控制
				* 16位校验和——校验TCP头部和数据，来可靠传输
				* 16位紧急指针
		* TCP状态转移
			* TCP连接的任一端在任意时刻都处于某种状态  可以通过netstat查看连接状态
			* 服务器端状态
				* 服务器通过LISTEN系统进入listen状态
				* 监听到连接请求，将连接放入内核等待队列，发送SYN确认报文段，处于SYN_RCVD
				* 收到确认报文段，处于established——此时可双向传输
				* 通过close主动关闭连接，返回确认报文段，处于close_wait
				* 服务器检测到客户端关闭后，也发送结束报文段，连接转移至LAST_ACK状态
			* 客户端状态
				* 主动与服务器连接，通过connect发送同步报文段，可能会如下失败
					* connect连接的目标端口不存在（未被任何进程监听）或者 端口仍被处于TIME_WAIT状态连接占用。connect调用失败，服务器返回复位报文段
					* 目标端口存在，但connect超时时间内未收到服务器确认报文。connect调用失败。返回初始close状态，调用成功则established状态
				* 客户端执行主动关闭，发送结束报文段。进入FIN_WAIT_1状态。 收到确认报文 处于FIN_WAIT_2状态。此时服务器处理close_wait
				* 收到服务器结束报文段后，确认并进入time_wait状态
			* TIME_WAIT状态
				* 客户端需要等待2MSL(报文段最大生存时间)才关闭，通常MSL是2min
				* 可靠的终止TCP连接
					* 因为确认服务器结束报文的最后一个报文段丢失，服务器会重发结束报文段，停留该状态可以处理重复冲到的结束报文段
				* 保证迟来的TCP报文有足够时间识别并丢弃
					* 因为同一个端口可能马上建立一个新的连接，这样可能受到之前的TCP连接报文段信息
					* 坚持2MSL正好确保两个传输方向上未收到的或者迟到的报文段都消失（生存时间到，被中转路由丢弃）
		* 复位报文段  ——携带RST的报文段  通知对方关闭连接或者重新建立连接 ：3种产生情况
			* 访问不存在端口  或者 端口被TIME_WAIT状态连接所占
			* 异常终止连接  socket自带SO_linger发送复位报文段异常终止连接
				* 异常终止连接->发送端所有排队等待发送的数据都将被丢弃
			* 处理半打开连接
		* TCP数据流——以长度区分
			* 交互数据——少字节，实时要求高，如telnet、ssh
				* 服务器采用延迟确认，在回复确认报文段时携带数据一起发送
			* 成块数据——大字节，传输效率高，如ftp
				* 发送方连续发送多个TCP报文段，接收方一次性确认。
				* 拒上次确认能连续发送多少个，由接收窗口（及拥塞窗口）大小决定
		* TCP超时重传——TCP模块为每个TCP报文段维护一个重传定时器
			* 每次超时重传时间增加一倍  与TCP超时重连策略一致
			* 5次重传均失败后 底层IP和ARP接管
		* TCP拥塞控制——提高网络利用率，降低丢包率，保证网络资源对每条数据流公平。
			* 拥塞控制算法在Linux下有多种实现，比如reno算法、vegas算法和cubic算法等。
			* 拥塞控制的是发送端向网络一次连续写入的数据量——SWND(send window 发送窗口），SWND限定了TCP报文段数量，这些报文段最大长度为SMSS(值一般等于MSS)
				* 接受方可以通过RWND(接受通告窗口)来控制发送方的SWND，SWND太小会有网络延迟，太大会导致网络拥塞。 发送端引入CWND拥塞窗口变量（congestion window).。实际SWND是 CWND和RWND的较小值
			* 拥塞控制算法——TCP连接建立后，CWND设置为1W(initial window) ,大小为2-4个SMSS
				* 判断拥塞发生的依据
					* 发送端
						* 传输超时，或者TCP重传定时器溢出
						* 接受的重复的确认报文段
				* 主动积极避免阶段——传输超时也是这两方案
					* **慢启动 slow start
						* 发送端每收到一个确认，进行指数增加。慢启动不慢，很快膨胀
						* 设置慢启动阈值 ssthresh, 当CWND大于阈值时，进入拥塞避免
					* **拥塞避免  congestion voidance
						* 按照线性增长，减缓其扩大
				* 拥塞发生后
					* 发生拥塞——当检测到传输超时
						* CWND调整小于SMSS
						* ssthresh 调整至不大于2倍SMSS  、慢启动阈值设置为当前窗口大小
						* 再次进入慢启动
					* 发生拥塞——连续收到3个重复ACK（比如TCP报文段丢失 或者 收到乱序报文段
						* **快速重传 fast retransmit 和 快速恢复 fast recover
							* 收到第三个重复ACK时
								* 慢启动阈值设置为当前窗口大小的一半
								* 立即重传报文段
								* 发送窗口大小设置为慢启动阈值加上3个MSS。
								* 进入拥塞避免 线性增长
							* 每收到一个重复ACK
								* 设置CWND为原CWND+SMSS
								* 此时发送端可以发送新的报文（在CWND允许的情况
							* 当收到新的ACK确认时
								* 设置ssthresh为CWND

		* telnet协议，远程登录协议

		* DNS协议。提供域名到IP地址的转换。
	* HTTP协议
		* http请求
			* 等幂——多次连续重复的请求是一样的结果
				* GET——申请获取服务器资源
				* HEAD——仅获取头部信息
				* OPTIONS——查看服务器对某个特定URL支持哪些请求方法
				* TRACE——返回原始HTTP请求内容
				* PUT——上传数据
				* DELETE——删除数据
			* 非等幂——请求影响了服务器资源
				* POST——提交数据
		* HTTP应答
			* 1** 信息
				* 100 Continue  服务器告诉客户端继续发送数据
			* 2** 成功
				* 200 OK  请求成功
			* 3** 重定向
				* 301  Move permanently  资源被转移，请求重定向
				* 302 Found  通知客户端资源能在其他地方找到，但需要GET方法获取
				* 304 Not modifed 被申请的资源没更新，和之前相同
				* 307 Temporary redirect  资源能在其他地方找到，可以使用原始请求方法获取
			* 4** 客户端错误
				* 400 bad request  客户请求错误
				* 401 unauthorized 用户需要认证信息
				* 403 forbidden  访问被服务器禁止
				* 404 Not found 资源没找到
			* 5** 服务器错误
				* 500 internet server error 服务器错误
				* 503 service unavailable 服务器暂时无法访问
		* Cookie
			* 因为HTTP是无状态协议，cookie来保持连接状态
			* 客户端每次给服务器发起HTTP请求 头部都携带cookie，这样服务器就能区别客户

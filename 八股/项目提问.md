## 线程池行为
**亮点**：线程池避免了线程频繁创建和销毁的开销，提高了资源利用率和响应速度。
##### 工作线程一直等待吗
在任务队列为空时会进入等待状态，通过条件变量进行等待的，当有新任务进入队列时，线程会被唤醒来执行任务
#### 线程池工作线程处理完一个任务后的状态是
完成任务后，工作线程会检查任务队列是否还有待处理任务，如果有继续执行新任务。
如果没有，线程返回等待状态，直到新任务到来或线程池关闭
#### 如果同时1000个客户端访问请求，线程数不多，怎么及时响应
使用事件驱动epoll配合线程池可以高效处理大量并发连接。
epoll可以高效管理多个网络连接的IO事件，只环形有数据或请求的连接
线程池可以并发处理这些请求，优化资源使用
#### 如果一个客户请求占用线程很多时间，会不会影响，有什么策略
会影响。为避免长任务阻塞线程，可以实施任务优先级，将长任务分解或异步处理，使用更多工作线程 或者专门线程处理长任务也是常见策略

**线程池如何提高Web服务器的性能？**
线程池通过重用一定数量的线程来处理请求，避免为每个请求创建和销毁线程带来额外高开销。这样可以快速响应，提高了处理能力。

## 并发模型
#### 服务器用的并发模型
是的，使用的Reactor模型，结合epoll作为IO多路复用技术，通过非阻塞IO高效处理并发事件
#### reactoe、paoactor、主从reactoe模型区别
* reactor：同步IO、主线程只负责监听文本描述符、将其派给相应处理程序
* proactor：异步IO、所有IO操作都由主线程异步执行，完成后通知应用程序
* 主从：主reactor负责监听连接请求，把连接分给从reactor，从reactor处理IO读写事件和请求，更好利用多核资源
#### 为什么用epoll，和其他复用方式的区别
epoll比select和poll有更高效的效率和拓展性，支持大数量并发连接，在处理活跃连接时不需要重复遍历。epoll支持ET边缘触发模式，在有感兴趣的事件发生后会立刻返回。 然后内核将这些就绪的文件描述符放到ready list中。*因为它只在状态改变时才通知应用程序，减少了事件触发的频率。*
select和poll使用固定数组大小，适合小并发。每次调用需要把文件描述符集合拷贝到内核态进行遍历，开销太大。而且只支持LT水平触发模式：遍历事件表来观察是否有感兴趣的事

**Epoll的水平触发与边沿触发有什么区别？**
水平触发下，只要被监听的文本描述符还有数据可读 事件就会触发。而边沿触发下只有数据状态发生变化，事件才会触发，这减少了事件触发频率

**如何理解非阻塞IO与epoll的结合使用？**
非阻塞允许程序在数据还未准备好的时候立即返回，继续执行其他任务。
epoll是IO多路复用技术，能让单个线程监听多个IO事件。
结合使用，可以让单线程内部处理多个并发的数据流，提高效率。

**线程环境中，如何保证数据一致性和线程安全？**
互斥锁 mutex 可以用来保护共享数据。确保任意时刻 只有一个线程访问这些数据。更复杂的数据结构操作， 可以使用更高级的同步机制，比如读写锁 shared_mutex等。


## HTTP报文解析
#### 为什么用状态机
因为他可以清晰的管理复杂的解析过程，将HTTP请求结构成多个明确的状态。通过状态机可以有效识别和响应不同部分的报文，比如请求行、头和消息体。这样能提高解析的效率和可靠性尤其处理不完整的请求更稳健
#### 为什么不用map
提高一个严格的方式来处理输入和转化状态，这对于解析协议非常合适，因为他可以精确地根据输入改变处理状态，优化解析过程，介绍出错率。相比map或者hash，状态机逻辑上更直接，执行效率更高
本质是 http协议头没有长度，所以接收完再解析效率慢，可以一行一行边接收边解析


## 日志
**亮点**：使用双缓冲区技术实现的异步日志系统，减少了日志写入对性能的影响
#### 日志系统的运行机制
一个前端收集日志信息，后端负责存储。
**日志系统通过将日志记录的写入操作放在一个单独的线程来实现，不阻塞主程序的执行。**
异步模式下，前端将日志消息放在一个线程安全的队列中，后端在另一个线程中从队列取出日志消息写入存储系统redis
#### 异步日志和同步的区别
同步：操作（写入） 在主程序流中执行，可能因为IO操作导致阻塞，影响性能
异步：操作在独立后台线程进行，主线程通过队列发送日志消息到后台线程，减少IO操作对主程序流影响，提高性能
#### 如果分发到不同机器
使用消息队列 如卡夫卡来收集分发日志。主服务器将日志发送到消息列队，其他机器订阅对应队列，实现日志的实时分发

**描述一个具体的内存泄漏问题，并说明如何发现和解决。**
可以使用工具Valgrind来检测内存泄露，当我发现泄露时，就立刻检查所有的new操作是否有对应的delete操作
* 在我的Web服务器项目中，我们使用了一个线程池来处理高并发请求，同时也使用了一个自定义的定时器来管理客户端连接的超时。在一次性能测试中，我们发现服务器在运行一段时间后，内存使用量不断增加，最终导致系统内存耗尽，服务器崩溃。
#### 问题描述
经过分析和调试，我们发现内存泄漏的来源是定时器管理代码中。具体来说，定时器在删除已超时的连接时，未能正确释放内存，导致内存泄漏
#### 如何发现内存泄漏
1.  **工具检测**：
    -   使用`valgrind`工具检测内存泄漏。运行服务器程序并使用`valgrind`进行检测，命令如下：
        `valgrind --leak-check=full --show-leak-kinds=all ./server`
2.  **日志记录**：
    -   在定时器管理代码中添加日志，记录每次添加和删除定时器的操作，检查是否有未删除的定时器。
3.  **监控内存使用**：
    -   使用系统监控工具（如`top`、`htop`）监控服务器的内存使用情况，观察内存使用量的变化。

#### 问题定位
通过`valgrind`工具的检测报告，我们发现定时器在删除超时连接时，没有正确释放`HeapTimer`对象。以下是问题代码的简化示例：
`void HeapTimer::add(int fd, int timeout, const std::function<void()>& cb) {     Node* node = new Node(fd, timeout, cb);     heap_.push(node);     fd_map_[fd] = node; }  void HeapTimer::del(int fd) {     auto it = fd_map_.find(fd);     if (it != fd_map_.end()) {         Node* node = it->second;         fd_map_.erase(it);         // 错误：没有删除heap_中的节点         // delete node;  // 应该在这里释放内存     } }`
在删除定时器时，只从`fd_map_`中删除了节点，但是没有从`heap_`中删除，也没有释放节点的内存，导致内存泄漏。
#### 解决方法
为了修复这个内存泄漏问题，我们需要确保在删除定时器时，正确释放`HeapTimer`对象的内存。修改后的代码如下：
`void HeapTimer::add(int fd, int timeout, const std::function<void()>& cb) {     Node* node = new Node(fd, timeout, cb);     heap_.push(node);     fd_map_[fd] = node; }  void HeapTimer::del(int fd) {     auto it = fd_map_.find(fd);     if (it != fd_map_.end()) {         Node* node = it->second;         fd_map_.erase(it);         auto heap_it = std::find(heap_.begin(), heap_.end(), node);         if (heap_it != heap_.end()) {             std::iter_swap(heap_it, heap_.end() - 1);             heap_.pop_back();             std::make_heap(heap_.begin(), heap_.end(), NodeCmp());         }         delete node;  // 释放内存     } }`
在删除定时器时，我们不仅从`fd_map_`中删除节点，还需要从`heap_`中找到并删除节点，同时释放节点的内存。

# 项目中遇到的难点是什么？怎么处理的  
### 挑战一：高并发处理
随着项目进入性能测试阶段，我发现服务器在面对高并发请求时性能出现了瓶颈，响应时间变长，偶尔还会发生连接丢失的情况。问题的根源在于服务器在高并发场景下无法有效处理大量的客户端连接，特别是在连接的建立和关闭时会有较大的性能开销。

### 解决方案：
1.  **优化事件处理模式：**
    -   我在服务器的事件处理模块中引入了边缘触发（Edge Triggered, ET）模式，与传统的水平触发（Level Triggered, LT）模式相比，边缘触发模式可以减少事件的重复触发，从而降低CPU的负载。为此，我在`WebServer`类中对监听事件和连接事件进行了相应的设置。
    -   代码示例如下：
        `void WebServer::InitEventMode_(int trigMode) {     listenEvent_ = EPOLLRDHUP;     connEvent_ = EPOLLONESHOT | EPOLLRDHUP;     if(trigMode == 1 || trigMode == 3) {         connEvent_ |= EPOLLET;     }     if(trigMode == 2 || trigMode == 3) {         listenEvent_ |= EPOLLET;     }     HttpConn::isET = (connEvent_ & EPOLLET); }`
        
    -   这样一来，服务器在处理大量连接时，CPU的使用效率得到了显著提升。
-   **引入线程池优化请求处理：**
    -   我使用了线程池来管理工作线程，通过将请求分发给线程池中的线程处理，减少了频繁创建和销毁线程的开销，提高了服务器的并发处理能力。
    -   例如，在`WebServer`类的`DealRead_`和`DealWrite_`方法中，我将请求处理任务提交给线程池：
        `void WebServer::DealRead_(HttpConn* client) {     assert(client);     ExtentTime_(client);     threadpool_->AddTask(std::bind(&WebServer::OnRead_, this, client)); }`
        
-   **使用数据库连接池提升数据库访问性能：**
    -   在项目中，我还实现了一个数据库连接池，管理与MySQL数据库的连接，避免了每次请求都要创建和销毁数据库连接的开销，显著提升了数据库的访问性能。
    -   连接池的实现如下：
        `void SqlConnPool::Init(const char* host, int port,             const char* user, const char* pwd, const char* dbName,             int connSize) {     for (int i = 0; i < connSize; i++) {         MYSQL* sql = mysql_init(sql);         sql = mysql_real_connect(sql, host, user, pwd, dbName, port, nullptr, 0);         connQue_.push(sql);     }     MAX_CONN_ = connSize;     sem_init(&semId_, 0, MAX_CONN_); }`

在高并发场景下，日志系统也成为了性能瓶颈之一。特别是同步日志记录会阻塞主线程，影响整体系统的吞吐量。

# 具体遇见了哪些问题？怎么定位排查和解决的
在我开发的高性能Web服务器项目中，遇到了几个非常棘手的问题，特别是在处理高并发请求时，服务器偶尔会崩溃，且响应时间不稳定。我通过多种手段进行了问题的定位和解决。
### 问题一：内存泄漏导致的服务器崩溃
**问题描述：** 在一次压力测试中，我发现服务器在长时间运行后会发生崩溃。通过分析，发现崩溃的原因是由于内存泄漏。最初我怀疑是某些动态分配的内存没有及时释放，但由于代码量较大，难以直接定位到问题所在。

**错误定位：** 我首先使用了`valgrind`工具来检测内存泄漏情况，结果显示在`HttpConn`类的`process()`方法中有泄漏。这一行代码引起了我的注意：
`if(readBuff_.ReadableBytes() <= 0) {     return false; }`

进一步分析代码，我发现问题出在`HttpConn::process()`方法内部，具体是在调用`response_.MakeResponse()`生成响应时，没有正确处理内存映射文件的释放。因为我在`HttpResponse::AddContent_()`中使用了`mmap`进行文件映射，但在处理完请求后，并没有及时调用`munmap`来释放映射的内存。

`void HttpResponse::AddContent_(Buffer& buff) {     int srcFd = open((srcDir_ + path_).data(), O_RDONLY);     if(srcFd < 0) {          ErrorContent(buff, "File NotFound!");         return;      }     mmFile_ = (char*)mmap(0, mmFileStat_.st_size, PROT_READ, MAP_PRIVATE, srcFd, 0);     close(srcFd);     buff.Append("Content-length: " + to_string(mmFileStat_.st_size) + "\r\n\r\n"); }`

**解决方案：** 为了解决这个问题，我在`HttpResponse`类的析构函数中添加了`UnmapFile()`方法，用于释放内存映射文件：

`HttpResponse::~HttpResponse() {     UnmapFile(); }  void HttpResponse::UnmapFile() {     if(mmFile_) {         munmap(mmFile_, mmFileStat_.st_size);         mmFile_ = nullptr;     } }`

通过这次修复，我不仅解决了内存泄漏的问题，还学习到了使用`mmap`进行内存映射时需要非常谨慎，必须在合适的时机释放内存。

### 问题二：边缘触发模式下的读写事件丢失
**问题描述：** 在项目中，我采用了Epoll的边缘触发（ET）模式来提高性能。然而，在高并发场景下，部分客户端的请求未能被正确处理，导致连接丢失。这主要发生在处理读事件时，有些数据未能被完全读取就导致连接关闭。

**错误定位：** 我仔细检查了`WebServer`类的`DealRead_()`和`DealWrite_()`方法，发现问题出在以下代码中：
`ssize_t len = readBuff_.ReadFd(fd_, saveErrno); if (len <= 0) {     break; }`

在ET模式下，必须将文件描述符中的数据全部读取完毕，但这里的逻辑在`read()`返回`EAGAIN`或读取到EOF后便立即停止读取，而实际上可能还有未读完的数据，导致数据丢失。

**解决方案：** 我调整了`read()`的逻辑，在ET模式下使用循环读取直到`read()`返回`EAGAIN`，确保所有数据都被读取：

`ssize_t HttpConn::read(int* saveErrno) {     ssize_t len = -1;     do {         len = readBuff_.ReadFd(fd_, saveErrno);         if (len <= 0) {             break;         }     } while (isET);     return len; }`

通过这一改动，成功解决了ET模式下读写事件丢失的问题。这个过程中，我深入学习了Epoll的工作原理和ET模式的使用技巧。

### 问题三：线程池任务调度不均衡

**问题描述：** 在高并发情况下，线程池中的某些线程负载很重，而其他线程几乎没有任务，导致整体性能不均衡。这个问题导致了一部分请求处理速度很慢。

**错误定位：** 我检查了线程池的任务分发逻辑，发现在高并发场景下，某些任务由于队列阻塞而无法及时分发给空闲线程。任务被集中到某些线程，而其他线程处于空闲状态。

**解决方案：** 为了均衡负载，我对任务调度策略进行了优化，改用公平锁（Fair Lock）来确保任务在线程间的均匀分配。同时，我在任务队列为空时，让线程进行自旋等待而不是直接阻塞，以便能够快速响应新的任务：

`void ThreadPool::AddTask(F&& task) {     {         std::lock_guard<std::mutex> locker(pool_->mtx);         pool_->tasks.emplace(std::forward<F>(task));     }     pool_->cond.notify_one(); }`

通过这些改进，线程池的负载分配变得更加均衡，整体性能得到了显著提升。这次调优让我更加了解了多线程调度和锁的使用，也提高了我对并发编程的理解。


##### 你提到的自旋锁和公平锁是什么？ 如何应用到项目中的
### 1. **公平锁（Fair Lock）**
公平锁是一种确保所有等待锁的线程按顺序（通常是FIFO）获取锁的机制，避免“饥饿”现象（即某些线程长时间得不到锁）。在标准的C++中，没有直接的“公平锁”概念，但我们可以通过条件变量和一个队列来实现类似的效果。

#### 示例代码：

`class FairLock { public:     FairLock() : next_ticket(0), current_ticket(0) {}      void lock() {         std::unique_lock<std::mutex> lock(mtx_);         size_t my_ticket = next_ticket++;         while (my_ticket != current_ticket) {             cond_.wait(lock);         }     }      void unlock() {         std::lock_guard<std::mutex> lock(mtx_);         current_ticket++;         cond_.notify_all();     }  private:     std::mutex mtx_;     std::condition_variable cond_;     size_t next_ticket; // 分配给下一个线程的票据     size_t current_ticket; // 当前正在执行的线程的票据 };`

#### 应用解释：

-   每个线程在请求锁时，都会得到一个“票据”，并且它会等待直到轮到它的“票据”才能获取锁。
-   当锁被释放时，下一个票据的线程会被唤醒并获取锁。这种机制确保了锁的公平性，所有线程都会按顺序获得锁，避免了某些线程一直无法获取锁的问题。

### 2. **自旋锁和自旋等待**

自旋锁（Spin Lock）是一种在等待锁时，不通过阻塞线程，而是通过持续尝试获取锁的方式来等待锁释放的机制。这在多核处理器中很有效，因为线程无需阻塞和唤醒的上下文切换开销。

#### 示例代码：

`class SpinLock { public:     void lock() {         while (lock_.test_and_set(std::memory_order_acquire)) {             // 自旋等待，忙等待直到锁被释放         }     }      void unlock() {         lock_.clear(std::memory_order_release);     }  private:     std::atomic_flag lock_ = ATOMIC_FLAG_INIT; };`

#### 应用解释：

-   自旋锁在等待锁时不会使线程进入睡眠状态，而是一直循环检查锁是否被释放。这种方法避免了线程的阻塞和上下文切换开销。
-   自旋锁适合短时间持有锁的场景，如果锁的持有时间较长，自旋锁可能会浪费CPU资源。

### 应用到你的项目中

假设你在`ThreadPool`中使用自旋等待和公平锁来优化线程池的任务调度，具体实现可能如下：


`class ThreadPool { public:     explicit ThreadPool(size_t threadCount = 8) : pool_(std::make_shared<Pool>()) {         assert(threadCount > 0);         for (size_t i = 0; i < threadCount; i++) {             std::thread([pool = pool_] {                 while (true) {                     std::function<void()> task;                     {                         FairLock lock(pool->mtx);                         while (pool->tasks.empty() && !pool->isClosed) {                             pool->cond.wait(lock);                         }                         if (pool->isClosed && pool->tasks.empty()) break;                         task = std::move(pool->tasks.front());                         pool->tasks.pop();                     }                     if (task) {                         task();                     }                 }             }).detach();         }     }      template<class F>     void AddTask(F&& task) {         {             std::lock_guard<FairLock> locker(pool_->mtx);             pool_->tasks.emplace(std::forward<F>(task));         }         pool_->cond.notify_one();     }  private:     struct Pool {         FairLock mtx;         std::condition_variable cond;         bool isClosed = false;         std::queue<std::function<void()>> tasks;     };     std::shared_ptr<Pool> pool_; };`

### 解释：

-   **公平锁**：在`ThreadPool`中使用`FairLock`替代了标准的`std::mutex`，确保任务在多个线程之间公平分配。线程进入队列并按照顺序获取锁，以保证负载均衡。
-   **自旋等待**：在这个示例中没有直接应用自旋锁，因为在线程池任务调度中，自旋锁可能导致不必要的CPU占用。如果你希望在某些关键的短期锁定场景中使用自旋锁，可以根据具体需求进行调整。

示例思路：
-   守护线程通过定期监控日志线程的心跳信号（例如定时更新的状态标志），判断日志线程是否运行正常。如果长时间未更新心跳信号，守护线程就会认为日志线程已经出现问题，并尝试重新启动。
- ### **日志持久化与恢复机制**
-   **临时缓冲区**：如果日志线程崩溃，在恢复日志线程时，未处理的日志数据可能会丢失。为防止这种情况，可以使用**持久化缓冲区**（例如文件、磁盘上的缓存）来存储未处理的日志，确保日志线程崩溃后能够恢复未写入的数据。
    -   在每次写入之前，将日志数据暂时保存在持久化介质中，保证即使日志线程崩溃或机器重启，也不会丢失数据。
-   **日志重放机制**：在日志线程重启时，可以从持久化缓冲区中读取未完成的日志任务，进行重放处理，确保日志数据完整性。

### JSON是一种轻量级的数据交换格式，采用纯文本表示
* JSON 序列化就是将**数据对象**转换成JSON字符串
* 因为 JSON 本身就是个 KV 结构，很容易映射到类似 map 的关联数组操作方式
* JSON for Modern C++ 使用一个 json 类来表示 JSON 数据
	* 添加完之后，用成员函数 dump() 就可以序列化
	* 反序列化功能同样也很简单，只要调用静态成员函数 parse() 就行，直接得到 JSON 对象，而且可以用 auto 自动推导类型：
* 递归下降JSON解析器和生成器，支持UTF-8编码。
* 选择UTF-8的原因
	* **它采用字节为编码单元，不会有字节序（endianness）的问题。**
	* **每个 ASCII 字符只需一个字节去储存**
	* **如果程序原来是以字节方式储存字符，理论上不需要特别改动就能处理 UTF-8 的数据。**
	* JSON 库不会校验 JSON 文本是否符合 UTF-8
* Document Object Model（DOM）是一个储存于内存的 JSON 表示方式，用于查询及修改 JSON。
	* 可以解析一个 JSON 至 DOM，然后就可以轻松查询及修改 DOM，并最终转换回 JSON。
	* 那么现在该 JSON 就会被解析至 `document` 中，成为一棵 _DOM 树_:
	* DOM 易于查询及修改。

#### 由于 TCP 是流式传输，且不会像 HTTP 那样自带消息边界，所以你需要定义自己的协议，确保每次传输的数据包都能被正确识别。可以采用以下方案：
* #### (1) **连接复用与并发处理**
TCP 长连接的一个典型问题是如何在一个连接上处理多个并发请求。因为 TCP 是流式协议，你需要确保每个请求和响应之间有明确的边界。为此，可以考虑以下几种方法：
	-   **请求和响应的长度头**：在每个请求和响应的前面附加一个长度字段，指示该消息的长度。这样接收方可以通过读取该长度字段来正确解析消息。
	-   **消息边界标识符**：使用特定的标识符或分隔符（例如 `\r\n`）来划分每个请求和响应。
- #### (2) **心跳检测**
在 TCP 长连接中，可能会出现连接闲置的情况。为了检测和保持连接的活跃性，你需要实现 **心跳机制**。即定期发送心跳包，确保连接没有因为超时或网络故障断开。
	-   **客户端定期发送心跳**：客户端可以在一定的时间间隔内向服务器发送心跳包。
	-   **服务器回应心跳**：服务器接收到心跳包后应该立即响应，表示连接仍然保持有效。
#### （3）**负载均衡与故障切换**
当系统规模变大，单一长连接可能会成为瓶颈，特别是当有多个服务器节点时，你可能需要考虑 **负载均衡** 和 **故障切换**。负载均衡可以让多个长连接分布在不同的服务器上，提高系统的吞吐量和可靠性。
-   **客户端连接池**：客户端可以维护一个连接池，每个连接与不同的服务器通信，从而实现负载均衡。
-   **故障切换**：当服务器节点出现故障时，客户端应能够自动切换到其他可用节点。
- #### (5) **多线程处理**
正如之前讨论的，在长连接的情况下，多个线程可能需要并发处理同一个连接上的多个请求。在这种情况下，你需要仔细管理多线程的并发问题，可以采用锁、队列或其他同步机制来确保并发安全。
* **brpc**：brpc 支持长连接，能够在长连接上处理大量并发请求，并且有负载均衡、故障检测和恢复机制。


#### `ProtoBuf` 与 `json` 的优缺点对比
-   `json` 没有类型，而 `protobuff` 是具有类型的，更好的融入编程语言
-   `json`是文本传输，`protobuff` 是二进制传输，后者解析速度更加快，序列化数据非常简洁、紧凑。
